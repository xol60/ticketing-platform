System architecture

Four-tier topology. Browser → nginx (edge) → API gateway → 9 Spring Boot services that communicate exclusively through Kafka → shared Postgres + Redis storage.

PropertyWhy
No HTTP between services for workflows Downed service degrades into consumer lag, not cascading 5xx
9 logical Postgres DBs on one master Bounded-context isolation without paying for 9 physical clusters
Redis is acceleration, not source of truth Saga state writes Postgres first; Redis outage = slow reads, not data loss
Stateless JVM State lives in Postgres / Kafka / Redis; pods can scale or restart freely

Flow 2 — Accept new price

Surge moved between orderCreatedAt and the pricing lock. Pricing emits PriceChangedEvent, saga pauses in AWAITING_PRICE_CONFIRMATION. User confirms via REST → saga re-issues lock command at the new price → resumes happy path.

Flow 3 — Decline new price (or 30-second timeout)

Same divergence as Flow 2 up to AWAITING_PRICE_CONFIRMATION. User rejects (or SagaWatchdog timeout fires). Saga compensates by releasing the ticket and emits OrderCancelledEvent. Order ends as CANCELLED; ticket returns to AVAILABLE; no payment row exists.
