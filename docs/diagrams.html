System architecture

▶️ Looking for animations? See the animated saga flows — interactive play/pause/step demo of all three scenarios.

Four-tier topology. Browser talks to nginx; nginx routes either to the API gateway (REST + SSE) or directly to the React SPA. All nine services communicate exclusively through Kafka — no service-to-service HTTP for business workflows.

EdgeNginx — rate-limit, gzip, SSE buffering off
GatewayJWT validation, circuit breaker, rate-limiter
Application9 Spring Boot services, stateless at JVM level
StorageKafka + Postgres master/slave + Redis

Notable design properties

PropertyWhy
No HTTP between services for workflows A downed service degrades into consumer lag, not cascading 5xx
9 logical Postgres DBs on one master Bounded-context isolation without paying for 9 physical clusters
Redis is acceleration, not source of truth Saga state writes Postgres first; Redis outage = slow reads, not data loss
Single read replica, all services Enough for current load; partition by service when any DB outgrows master

Flow 1 — Happy path (price unchanged)

User's requestedPrice matches facePrice × surge_at_orderCreatedAt. Saga walks all 10 hops without compensation and terminates in COMPLETED.

Flow 2 — Price changed, user accepts new price

Surge multiplier moved between orderCreatedAt and the pricing lock. Pricing service emits PriceChangedEvent, saga pauses in AWAITING_PRICE_CONFIRMATION. User confirms via REST → saga re-issues PriceLockCommand confirmed=true at the new price.

    OS ->> K:  order.created
    OS -->> U: 201 PENDING

    K  ->> SO: OrderCreatedEvent
    SO ->> K:  ticket.cmd TicketReserveCommand
    K  ->> TS: reserve
    TS ->> DB: tickets RESERVED
    TS ->> K:  ticket.reserved

    K  ->> SO: TicketReservedEvent
    SO ->> K:  pricing.lock.cmd confirmed=false
    K  ->> PR: PriceLockCommand
    Note right of PR: Case C mismatch
but multiplier in history
    PR ->> K:  pricing.price.changed PriceChangedEvent

    K  ->> SO: PriceChangedEvent
    SO ->> DB: saga_states AWAITING_PRICE_CONFIRMATION
    SO ->> K:  order.price.changed
    K  ->> OS: OrderPriceChangedEvent
    OS ->> DB: orders PRICE_CHANGED pending_price=newPrice
    OS -->> U: SSE PRICE_CHANGED pendingPrice

    Note over U,SO: 30-second watchdog window
    U  ->> OS: POST /confirm-price
    OS ->> K:  order.price.confirm
    K  ->> SO: OrderPriceConfirmCommand

    SO ->> K:  pricing.lock.cmd confirmed=true
    K  ->> PR: PriceLockCommand
    PR ->> K:  pricing.locked newPrice

    K  ->> SO: PricingLockedEvent
    SO ->> K:  payment.cmd amount=newPrice
    K  ->> PS: charge
    PS ->> DB: payments SUCCESS
    PS ->> K:  payment.succeeded

    K  ->> SO: PaymentSucceededEvent
    SO ->> K:  ticket.cmd TicketConfirmCommand
    K  ->> TS: confirm
    TS ->> DB: tickets CONFIRMED
    TS ->> K:  ticket.confirmed

    K  ->> SO: TicketConfirmedEvent
    SO ->> K:  order.confirmed
    K  ->> OS: OrderConfirmedEvent
    OS -->> U: SSE CONFIRMED

Flow 3 — Decline new price (or 30-second timeout)

Same divergence as Flow 2 up to AWAITING_PRICE_CONFIRMATION. User rejects (or watchdog timeout fires). Saga compensates by releasing the ticket and emits OrderCancelledEvent. Order ends as CANCELLED; ticket returns to AVAILABLE; no payment row exists.

    SO ->> DB: saga_states CANCELLED
    SO ->> K:  order.cancelled
    K  ->> OS: OrderCancelledEvent
    OS ->> DB: orders CANCELLED
    OS -->> U: SSE CANCELLED
